# gd32/f1x0: add IWDG (FWDGT) support

This is a proposed libopencm3 patch that would add IWDG support to the
GD32F1x0 stub. **It has not been submitted upstream** — per regtrace's
"never auto-PR" policy, the patch lives here for human review before any
upstream submission.

## Status
Draft, generated by regtrace v0.2.

## What this changes
- Adds `include/libopencm3/gd32/f1x0/iwdg.h` as a one-line forwarder to
  `<libopencm3/stm32/common/iwdg_common_v2.h>`. No carve-out required.
- Adds a `GD32F1X0` branch to `include/libopencm3/stm32/iwdg.h`'s chip
  dispatcher (mirroring the existing `GD32F1X0` branches in `gpio.h`,
  `flash.h`, `rcc.h`, `memorymap.h`).
- Adds `iwdg_common_all.o` to `lib/gd32/f1x0/Makefile`'s OBJS list.

Three files, +41 / -0 lines.

## Why this is safe to share rather than fork

GD32F1x0 IWDG (vendor name FWDGT — Free Watchdog Timer) is
register-compatible with STM32F0 IWDG including the window-mode register
(WINR) and early-wakeup register (EWCR). Same KR magic constants
(0x5555 / 0xAAAA / 0xCCCC), same per-register layout, same bit positions.

This was verified empirically by regtrace v0.2 by capturing register-write
traces from both the GigaDevice SPL `fwdgt_config + fwdgt_enable` path
and the libopencm3 `iwdg_set_period_ms + iwdg_start` path with inputs
chosen to land on the same final IWDG_PR / IWDG_RLR values. Final-state
match.

Full evidence:
- `regtrace/decisions/v0.2/IWDG.md` — decision document
- `regtrace/golden/gd-spl/v3.2.0/gd32f1x0/iwdg_config_2sec_period.{trace,elf}`
- `regtrace/golden/libopencm3/v0.8.0-457-g7b6c2205/stm32f0/iwdg_config_2sec_period.{trace,elf}`
- `regtrace/vectors/iwdg/config_2sec_period.yaml` — the vector

## What this does not change
- No behavioural change to STM32 targets. The `stm32/iwdg.h` dispatcher
  edit is purely additive — the `GD32F1X0` branch is only taken when
  `-DGD32F1X0` is passed at compile time.
- No new functions, no API changes, no test changes.

## Worth noting in review
libopencm3's `iwdg_set_period_ms` writes `IWDG_KR = 0xCCCC` (start) at
the *top* of the function before the unlock key. gd-spl never does this.
The behavioural effect is to start the watchdog with reset values
(timeout ≈ 512 ms at 32 kHz LSI) before reconfiguring. Probably
intentional defense-in-depth in libopencm3, but worth being aware of when
porting code that assumes the gd-spl ordering. Not a bug; not a
portability concern; but a real difference any reviewer might wonder
about, so flagging.
