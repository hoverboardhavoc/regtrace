# gd32/f10x: initial port stub — TIMER + IWDG

This is a proposed libopencm3 patch that bootstraps the GD32F10x family
(F101 / F103 / F105 / F107) port by sharing STM32F1 common drivers for
the peripherals where empirical register-trace evidence proves the
silicon is identical. **It has not been submitted upstream** — per
regtrace's "never auto-PR" policy, the patch lives here for human
review before any upstream submission.

## Status
Draft, generated by regtrace v0.5. Builds cleanly locally; not yet
bench-validated on real GD32F103 silicon.

## What this changes

Eight files, +~150 / -0 lines:

- **New**:
  - `include/libopencm3/gd32/f10x/memorymap.h` — peripheral base addresses
    in STM32F1-style names so common drivers find them.
  - `include/libopencm3/gd32/f10x/timer.h` — one-line forwarder to
    `<libopencm3/stm32/common/timer_common_all.h>`.
  - `include/libopencm3/gd32/f10x/iwdg.h` — one-line forwarder to
    `<libopencm3/stm32/common/iwdg_common_all.h>`.
  - `include/libopencm3/gd32/f10x/rcc.h` — provisional forwarder to
    `<libopencm3/stm32/f1/rcc.h>` (see open questions below).
  - `include/libopencm3/gd32/f10x/doc-gd32f10x.h` — Doxygen group anchor.
  - `lib/gd32/f10x/Makefile` — minimal initial OBJS list:
    `timer_common_all.o` + `iwdg_common_all.o`.

- **Modified** (chip-dispatcher additions; STM32 targets unaffected):
  - `include/libopencm3/gd32/memorymap.h` — `+ #elif defined(GD32F10X)`
  - `include/libopencm3/stm32/memorymap.h` — `+ #elif defined(GD32F10X)`
  - `include/libopencm3/stm32/timer.h` — `+ #elif defined(GD32F10X)`
  - `include/libopencm3/stm32/iwdg.h` — `+ #elif defined(GD32F10X)`
  - `include/libopencm3/stm32/rcc.h` — `+ #elif defined(GD32F10X)`

## Why this is safe to share rather than fork

GD32F10x's TIMER and IWDG are register-compatible with STM32F1. Validated
empirically by regtrace v0.5 — final-state register traces match across
six independent oracles:
- libopencm3 stm32f0 / stm32f1 / stm32f4
- GigaDevice SPL gd32f10x / gd32f1x0
- STMicro Cube LL stm32f1

For both the TIMER and IWDG vectors, all 15 cross-implementation pairs
match in `final_state` mode. The actual byte-for-byte trace evidence is
in:

- `regtrace/decisions/v0.5/TIMER.md` — share-or-split write-up
- `regtrace/decisions/v0.5/IWDG.md` — share-or-split write-up
- `regtrace/golden/gd-spl/4ead0d8/gd32f10x/{timer,iwdg}.{trace,elf}` — gd-spl reference
- `regtrace/golden/libopencm3/v0.8.0-458-gd7e20526/stm32f1/{timer,iwdg}.{trace,elf}` — libopencm3 stm32f1 reference
- `regtrace/golden/cube-ll/v1.8.7-1-g5f2bc59/stm32f1/{timer,iwdg}.{trace,elf}` — Cube LL reference

End-to-end validation: regtrace was used to build a vector with this
patch's `libopencm3/gd32f10x` impl and compare it directly to the
`gd-spl/gd32f10x` impl. Result: **match in final_state mode for IWDG**.
The patch produces register-equivalent behaviour to the vendor SPL.

## Known limitations / what's not in this PR

- **Only TIMER + IWDG.** The other peripherals (GPIO, USART, DMA, I2C,
  ADC, SPI, FLASH) follow in subsequent PRs as their share-or-split
  decisions are made by regtrace. v0.6+ will add them.
- **rcc.h is provisional.** Forwarded to `stm32/f1/rcc.h` to unblock
  TIMER + IWDG compilation, but no regtrace RCC vector has been written
  yet. Code using clock-tree configuration (`rcc_clock_setup_*`) on
  GD32F10x should validate against gd-spl traces before relying on this
  forwarder.
- **No bench validation yet.** Register traces from a CPU emulator are
  necessary but not sufficient. The PR submitter should test on real
  GD32F103 silicon and report results in the PR body.

## Worth noting in review

- This patch is the second slice of GD32 port work informed by regtrace —
  the first was `regtrace/gd32f1x0-iwdg` (proposed for the F1x0 family).
  The patches are independent.
- The dispatcher edits in `stm32/{timer,iwdg,rcc,memorymap}.h` follow the
  pattern already established by the GD32F1X0 entries in `stm32/gpio.h`
  and the matching gd32 forwarders in `gd32/{memorymap,gpio,flash}.h`.
- The OBJS list is intentionally minimal. The expectation is that
  follow-up PRs add one or two peripherals at a time, each backed by a
  regtrace decision document and a vector that produces a matching trace.

## Submission checklist (for the human submitter)

- [ ] Confirm CLA / contributor agreement is up to date.
- [ ] Tag karlp on review since this opens a new GD32 family directory.
- [ ] Bench-validate on real GD32F103 silicon: configure a TIMER, observe
      PWM output on a scope; configure IWDG, intentionally starve the
      reload, confirm reset fires at the expected timeout.
- [ ] Mention regtrace evidence in the PR body so reviewers can reproduce.
- [ ] Be explicit that this is the FIRST slice — name the follow-up PRs
      planned (GPIO, USART, DMA, etc.) so the maintainer sees the full
      arc.
- [ ] Do **not** ask for the regtrace repo to be cited as a build-time
      dependency. The patch is self-contained.
